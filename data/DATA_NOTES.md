# Data notes — nyc_venues.json

## What this is
A **seed dataset** of 104 real NYC venues for local development (milestones
M1–M3). Generated by `scripts/build_dataset.py`. Schema: `docs/DESIGN.md §4`.

## Composition
- **68 dinners** across tiers: 13× $ / 11× $$ / 40× $$$ / 40× $$$$
- **19 drinks** (cocktail / wine / hotel-bar-view)
- **8 dessert**, **9 walk** pseudo-venues
- Michelin: 4× 3-star, 9× 2-star, 15× 1-star
- 6 walkable clusters (West Village, EV/LES, Flatiron/Gramercy, Midtown,
  Brooklyn, Tribeca/Soho) — each has both dinners and pre/post options so
  evening plans can always be assembled.

## How tags were derived
Each venue inherits a base tag profile from an **archetype** (e.g.
`omakase_counter`, `lively_bistro`, `hotel_bar_view`) defined in the generator,
plus per-venue overrides. This keeps all rubric axes on a consistent 0–1 scale.
Tags are informed estimates, not measured — they're meant to be refined.

## ⚠️ Verification status — `"verified": false` on every record
Venue **facts** come from general knowledge and must be verified before any
live/user-facing use (milestone M4 enrichment + spot-check):
- **Michelin status** changes yearly — re-check against the current guide.
- **Booking platform / `requires_deposit`** — confirm Resy vs OpenTable vs Tock;
  `url_template` is `null` and must be filled per platform (see `docs/DESIGN.md §5`).
- **lat/lng** are approximate — geocode properly before computing walk distances.
- **Tag values** — spot-check the subjective axes against real reviews.

## Fixing coordinates (geocoding)
`scripts/geocode.py` resolves accurate lat/lng and flips `verified:true`
(default provider: Photon, keyless). Run locally with network:
```
python3 scripts/geocode.py --dry-run   # preview
python3 scripts/geocode.py             # write in place
```
Large jumps from the seed point are flagged and NOT auto-written — review by hand.

## Photos (Google Places)
`scripts/fetch_photos.py` fills `photo_url` + `photo_credit` from the Google
Places API (official provider photos) and downloads them into `frontend/photos/`.
Needs `GOOGLE_MAPS_API_KEY` (Places API enabled + billing). Cards show the photo
when present, else the illustrated route map. **Licensing:** Google restricts
caching Place photos (~30 days) and requires showing attribution (we store/show
`photo_credit`) — fine for a personal/demo project; use a server-side proxy for
anything public/long-lived. Run:
```
export GOOGLE_MAPS_API_KEY=...
python3 scripts/fetch_photos.py --dry-run   # preview
python3 scripts/fetch_photos.py             # download + write
```

## Regenerate
```
python3 scripts/build_dataset.py
```
Note: regenerating overwrites geocoded coords (the generator is the dinner/venue
source of truth). Geocode after regenerating, or fold verified coords back into
the generator's `V` list.
Edit the `V` list (and archetype profiles `A`) in the script, not the JSON, so
the schema stays consistent. The JSON is the build output; the script is source.
